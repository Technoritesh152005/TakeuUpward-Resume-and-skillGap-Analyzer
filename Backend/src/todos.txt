1. What problem he is talking about

You currently:

Frontend → 3 API calls (one for each role)

👉 This is bad because:

slow ❌
heavy on server ❌
not scalable ❌
2. What he suggests (main idea)

👉 Do NOT call API 3 times from frontend

Instead:

Frontend → 1 API call → Backend handles everything
3. Simple flow (VERY IMPORTANT)
❌ Your current way
Frontend:
  call API (role1)
  call API (role2)
  call API (role3)
✅ Better way (his suggestion)
Frontend:
  send ONE request

Backend:
  → get resume
  → get all roles
  → process all
  → return combined result
4. Example
Frontend sends:
{
  "resumeId": "123",
  "jobRoleIds": ["1", "2", "3"]
}
Backend does:
Load resume once
Load all roles in one DB query
Run comparison for each role
Return all results together
5. Why this is better

👉 Instead of:

3 network calls ❌

👉 You get:

1 network call ✅
faster response
less load
6. Important improvements he mentioned
🔹 1. Concurrency control

👉 Don’t run 20 AI calls at once

Use limit:

Only 2–4 at a time
🔹 2. Caching (VERY IMPORTANT)

👉 If same resume + role:

Don’t recompute → return stored result
🔹 3. Preprocessing

👉 Extract resume skills once

❌ Not:

Extract skills again for every role
🔹 4. Queue system (ADVANCED)

If too many roles:

→ Put job in background
→ Process slowly
→ frontend waits/polls
🔹 5. Two-layer system
Fast logic → instant result
AI → deeper analysis later
7. Final simplified idea

👉 He is saying:

Don’t make multiple frontend API calls
→ Do everything in backend in one request
→ Optimize with caching + concurrency + queue
8. One-line summary (VERY IMPORTANT)

👉 Move complexity from frontend to backend

9. Why this is strong (interview level)

If you say this:

"I reduced multiple API calls into a single aggregated backend request with controlled concurrency and caching"

🔥 That’s very strong answer
